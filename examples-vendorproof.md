# VendorProof HOA — Sample Report Data (example.md)

**Purpose:** Three fully specified sample projects for the landing page "View Sample Report" asset and pilot demos. All community names, vendors, and figures are **fictional but market-accurate** for the Sarasota–Bradenton–Venice metro.

**Hard rule:** Every rendered report must carry the footer disclaimer:

> *Sample report — illustrative project data. Communities, vendors, and figures shown are fictional.*

Do not use real community or management company names anywhere in samples.

---

## Report Template Structure (applies to all three)

Every board-ready report renders these sections in order:

1. **Project Overview** — community, vendor, project name, contract dates, manager of record
2. **Budget Summary** — approved budget, contract amount, billed to date, paid to date, remaining, variance
3. **Billing vs. Progress Indicator** — % billed vs. % documented complete, status color (Green / Yellow / Red)
4. **Milestone Status Table** — milestone, expected payment %, due date, status, photo count
5. **Photos by Milestone** — grid, timestamped, grouped per milestone (placeholders in samples)
6. **Invoice / Payment Request Status**
7. **Change Order Log**
8. **Open Issues**
9. **Manager Recommendation**
10. **Next Steps**
11. Footer disclaimer (samples only)

**Status logic (for the indicator in section 3):**
- **Green:** billed % ≤ documented completion % + 10 points
- **Yellow:** billed % exceeds documented completion % by 11–25 points, OR required documentation missing on an invoiced milestone
- **Red:** billed % exceeds documented completion % by >25 points, OR invoice submitted against a milestone with zero documentation

## Implementation Notes for CTO

### Data model implications (fields every sample exercises)
- `projects`: community, location, community_type, project_name, vendor, contract_date, target_completion, manager_of_record, approved_budget, contract_amount, retainage_pct
- `milestones`: sequence, name, payment_pct, due_date, status (`complete | in_progress | invoiced_docs_incomplete | pending | not_started`), photo_count
- `invoices`: number, milestone_id, amount, received_date, status (`paid | hold | under_review | recommended`)
- `change_orders`: number, description, amount, status (`approved | pending_board | unapproved_submitted`)
- `open_issues`: free text, ordered
- `report`: computed billed_pct, documented_completion_pct, status_color, manager_recommendation (free text), next_steps (checklist)

### Status color computation
Implement the Green/Yellow/Red logic from the template section above as a pure function — it drives the single most demo-able element of the report (the Billing vs. Progress indicator).

### Rendering priorities
1. **PDF export of Sample 1 (Cypress Landing) is the landing page asset.** Build that renderer first.
2. Photo grids use placeholder images in samples — grey tiles with timestamp overlay ("Apr 3, 2026 • 8:42 AM • Milestone 1") sell the timestamping feature without real photos.
3. Footer disclaimer is mandatory on all three sample renders.
4. Landing page order: Sample 1 (catch a problem) → Sample 2 (regulatory/SIRS) → Sample 3 (clean close-out).

### Copy hooks tied to each sample
- Sample 1: "Know when billing runs ahead of the work."
- Sample 2: "Board-ready documentation for SIRS-driven capital projects."
- Sample 3: "Good contractors get approved — and paid — faster."
